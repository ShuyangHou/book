# 四、第二阶段：处理“一个实例覆盖多本书”的问题

BoT-SORT 只能关联已有实例，不能把一个错误的大实例自动拆成多本书。

因此，在实例进入跟踪器之前，需要增加异常实例检查。

## 1. 标记疑似合并实例

以下情况可以标记为：

```text
suspected_merged_instance
```

判断条件包括：

* 实例宽度明显大于当前帧书脊宽度中位数；
* mask 横向跨度异常；
* 一个实例区域中存在多个明显书脊边界；
* 当前大实例同时覆盖上一帧多个独立轨迹；
* 前后帧中该区域被分成多本书，而当前帧突然合并。

宽度只作为候选判断，不能因为一本书较宽就直接强制拆分。

例如：

```text
当前实例宽度 > 当前帧书脊宽度中位数的 2 倍
→ 标记为疑似合并
```

## 2. 优先使用前后帧恢复

例如：

```text
第10帧：A、B、C 三本书分别识别
第11帧：A+B+C 被错误合并为一个大 mask
第12帧：A、B、C 又分别识别
```

这时第 11 帧的大实例不应创建一个新 book_id，也不能把三个轨迹合并。

应当：

```text
标记 merged_detection
→ 不使用它更新单一轨迹
→ 依靠前后帧继续维持 A、B、C 三条轨迹
```

## 3. 持续合并时进行局部重分割

若一个区域连续多帧都出现合并：

```text
疑似合并区域
→ 裁剪局部图像
→ 放大 1.5～2 倍
→ 再运行一次 MaskDINO
→ 将局部实例映射回完整帧坐标
```

如果局部推理得到多个合理实例，就替换原来的大实例。

如果仍然无法拆分，则标记：

```text
multi_book_review
```

进入人工复核，不直接将其视为一册。

# 五、第三阶段：接入官方 BoT-SORT

不再继续使用自研的相位相关、AKAZE、手写匈牙利匹配和自定义 Track 类。

统一使用：

```text
MaskDINO-R50
+
NirAharon/BoT-SORT
```

## 1. BoT-SORT 的输入

每一帧将 MaskDINO 实例转换为：

```text
[x1, y1, x2, y2, score]
```

这里的 bbox 不是额外目标检测模型生成的，而是由 MaskDINO 的实例 mask 计算出的外接矩形。

同时传入当前原始帧或 ROI 帧，用于全局相机运动补偿。

## 2. BoT-SORT 的输出

BoT-SORT 为同一本实体书分配稳定的：

```text
track_id
```

业务层转换为：

```text
book_id = <video_id>_book_<track_id>
```

例如：

```text
001_book_0027
```

## 3. 初始配置

第一版建议：

```text
sample_fps = 3
with_reid = False
cmc_method = orb

track_high_thresh = 0.50
track_low_thresh = 0.10
new_track_thresh = 0.60
match_thresh = 0.80
```

`track_buffer` 不建议直接使用 30。

因为在 3 FPS 下：

```text
track_buffer = 30
≈ 允许轨迹丢失10秒
```

时间过长，容易把后面相似的书错误连接到旧轨迹。

第一轮建议测试：

```text
track_buffer = 6～12
```

即大约保留 2～4 秒。

## 4. 必须保留原实例映射

需要明确记录：

```text
当前 track_id
对应 detections.csv 中的哪个 instance_id
对应哪个 mask
对应哪个 crop
```

不能只根据返回顺序猜测。

输出：

```text
track_observations.csv
```

字段至少包括：

```csv
book_id,video_id,track_id,frame_index,time_sec,instance_id,score,x1,y1,x2,y2,mask_path,raw_crop_path,masked_crop_path
```

# 六、第四阶段：轨迹状态判定与实体计数

不是 BoT-SORT 产生的每一个 track_id 都自动算作一本书。

轨迹状态建议分为四类。

## 1. confirmed

满足：

```text
至少出现 2 个采样帧
并且至少有一次完整进入 ROI
```

自动计入实体册数。

## 2. singleton_review

只出现一次：

```text
num_observations = 1
```

默认不自动计数，进入人工复核。

## 3. boundary_review

轨迹有多个观测，但始终只在画面或 ROI 边缘出现，没有完整书脊 crop。

不自动计数，进入人工复核。

## 4. invalid

包括：

* bbox 非法；
* crop 不存在；
* 面积过小；
* 明显误检；
* 疑似多本书合并且无法拆分。

不计数。

最终自动实体册数为：

```text
confirmed 轨迹数量
```

不能使用：

```text
最大 track_id
总轨迹数
总检测实例数
```

作为最终册数。

## 输出文件

```text
unique_books.csv
```

字段：

```csv
book_id,video_id,track_id,first_frame,last_frame,num_observations,status,count_as_book,best_crop_path
```

同时生成：

```text
tracked_video.mp4
```

要求同一本书在连续帧中保持相同 `book_id`。

# 七、第五阶段：为每个 book_id 选择最佳 crop

同一个 `book_id` 会对应多张书脊图像。

例如：

```text
001_book_0027
├── frame_010 crop
├── frame_011 crop
├── frame_012 crop
└── frame_013 crop
```

不应该只根据 MaskDINO 置信度选择 OCR 图片。

综合考虑：

```text
图像清晰度
分割置信度
书脊面积
mask占bbox比例
是否接触边缘
是否完整进入画面
是否存在运动模糊
```

质量分数可以设计为：

```text
quality_score =
0.35 × 清晰度
+ 0.25 × 分割置信度
+ 0.15 × 面积
+ 0.15 × mask完整度
+ 0.10 × crop完整度
- 边缘惩罚
```

每条轨迹建议保留质量最高的 2～3 张，而不是只留一张。

输出：

```text
output/jobs/001/books/
├── 001_book_0001/
│   ├── best_raw.jpg
│   ├── best_masked.jpg
│   └── candidate_02.jpg
├── 001_book_0002/
└── ...
```

其中：

* `raw crop` 保留少量背景，避免 mask 漏掉文字；
* `masked crop` 将 mask 外区域设为白色，减少相邻书脊干扰。

两种图都可以送入 OCR 比较结果。

# 八、第六阶段：接入 OCR

OCR 不只是识别书名，还参与后续计数校验。

## 1. OCR 基本流程

```text
一个 confirmed book_id
→ 取质量最高的 2～3 张 crop
→ 尝试原方向、顺时针90°、逆时针90°
→ 调用 DeepSeek-OCR
→ 保存各次识别结果
→ 进行轨迹内投票
```

输出：

```text
ocr_results.csv
```

字段：

```csv
book_id,crop_path,crop_type,rotation,ocr_text,api_status
```

## 2. 轨迹内 OCR 投票

例如：

```text
crop_1：现代物流企业经营管理
crop_2：现代物流企业经营管埋
crop_3：现代物流企业经营管理
```

最终轨迹 OCR 结果：

```text
现代物流企业经营管理
```

这样比只识别一张 crop 更稳定。

## 3. OCR 失败不删除实体书

若 OCR 失败：

```text
book_id 仍然存在
book_title = unknown
count = 1
```

因为“看不清书名”不代表“没有这本书”。

# 十、第八阶段：使用 OCR 辅助校验轨迹

实体计数的主要依据仍然是 `book_id`，但 OCR 可以辅助发现跟踪错误。

## 1. 修复轨迹断裂

假设同一本书被 BoT-SORT 分成两条轨迹：

```text
track_001：第1～6帧
track_047：第8～14帧
```

只有同时满足以下条件，才允许自动合并：

```text
两条轨迹没有在同一帧同时出现
时间上前后连续
书架位置接近
OCR匹配到同一标准书名
crop外观相似
宽高接近
```

不能仅因为 OCR 书名相同就合并，因为书架上可能有多册同名书。

## 2. 发现错误轨迹合并

如果同一条轨迹前后 OCR 结果明显不同：

```text
轨迹前半段：数据库原理
轨迹后半段：计算机组成原理
```

则可能发生：

```text
ID switch
或
两本书被错误合并
```

应标记：

```text
possible_id_switch
```

进入自动拆分或人工复核。

因此 OCR 的作用包括：

```text
识别书名
按书名汇总册数
辅助合并断裂轨迹
发现错误合并
辅助人工复核
```
